## 一、整体架构分层总览

```
┌──────────────────────────────────────────────────────────────────────────┐
│  Layer 6：交付与版本化层  (cmd/* + doc/*)                                │
│    cmd/v0.1启动脚本  cmd/v0.2启动脚本  版本化用户手册  设计文档            │

│    sprites  audio/bgm  audio/sfx  tilesets  ui                           │
└──────────────────────────────────────────────────────────────────────────┘
```

## 二、各层模块职责详解

---

### Layer 4：全局单例层（Autoload）

> **6 个单例**，均在 `project.godot:24~31` 注册，全项目可用全局别名（`Config` / `Flags` / `Saves` 等），生命周期 = 应用全程。

| 单例文件 | 全局别名 | 职责 | 关键对外接口 |
|---|---|---|---|
| `autoload/GameEvents.gd` | `GameEvents` | **全局事件总线**：解耦模块间直接引用，所有跨模块通知走信号 | `dialog_started` / `enemy_died` / `gold_picked` / `objective_done` 等20+信号 |
| `autoload/ConfigManager.gd` | `Config` | **四层配置体系**：L1硬常量→L2平衡→L3关卡→L4用户设置，高优先级覆盖低优先级；支持 dotPath 查询（`"player.jump_force"`） | `get(path, default)` / `get_layer(layer)` / `reload_user_settings()` / `dump()` |
| `autoload/ProgressFlags.gd` | `Flags` | **进度标志存储器**：分两套 API——布尔 `Set/Get/Has`（章节解锁/目标完成）+ KeyValue `SetKV/GetKV`（存字符串路径/槽位/数值）；自动序列化进存档 | `Set(key,b)` / `Get(key)` / `SetKV(k,v)` / `GetKV(k,def)` / `serialize()/deserialize()` |
| `autoload/SaveSlotManager.gd` | `Saves` | **3 槽存档管理**：读写 `%APPDATA%/Godot/app_userdata/MedievalRebellion/slot_{1,2,3}.sav` + settings.json；自动加时间戳+校验 | `NewGame(slot)` / `Save(slot)` / `Load(slot)` / `ListSlots()` / `SaveSettings()/LoadSettings()` |
| `autoload/InputBus.gd` | `InputBus` | **输入二次派发**：引擎 InputMap 先到这里，归一化键盘+手柄（XBox/PS布局兼容），统一发信号；带「按下/持续/松开」三种事件 + 冷却防抖 | `jumped` / `attacked` / `block_changed(pressing)` / `move_axis(vec2)` 等 |
| `autoload/LevelFlowController.gd` | `LevelFlow` | **关卡流程 7 状态 FSM**：LOAD→READY→PLAYING→PAUSED→WIN→LOSE→EXIT；注入目标系统（Objectives）、检查点、传送门触发；失败/胜利条件判定 | `current_state` / `start_level(map_data)` / `add_objective(def)` / `is_all_objectives_done()` / `change_state(to)` |

### Layer 3：领域脚本层

#### 3.1 `scripts/config/`（配置工具）

| 文件 | 职责 |
|---|---|
| `ConfigLayer.enum.gd` | 定义 `ConfigLayer` 枚举：`L1_CONSTANTS = 0, L2_BALANCE = 1, L3_LEVELS = 2, L4_USER = 3`，供 ConfigManager 内部索引 |

#### 3.2 `scripts/save/`（存档工具）

|---|---|

#### 3.3 `scripts/editor/`（V0.2 编辑器 + 工坊核心，14 个脚本）

| 文件 | 类别 | 职责 | 关键接口 |
|---|---|---|---|
| `CharacterBase.gd` | 角色基类 | 所有可受伤/可死亡角色（玩家/同伴/敌人）的统一父类，V0.3 角色系统前置；包含 HP/体力/硬直/击退/无敌帧/朝向 | `take_damage(dmg,source,opts)` / `heal(n)` / `die()` / 信号 `hp_changed`/`hit`/`died`/`face_changed` |
| `DrawTools.gd` | 绘制工具 | 编辑器中画布操作的核心：笔刷/橡皮/矩形填充 3 种工具；50 步撤销/重做栈（双数组）；TileMap 层与 cell 批量写入 | `set_tool(brush/eraser/rect)` / `apply(pos)` / `undo()` / `redo()` / `get_stack_depth()` |
| `TilePalette.gd` | 左面板·图块 | 8 层切换（L0天空→L7前景）+ 5 套图集选块（Farm/Village/Forest/Castle/Admin，ID区间 1/101/201/301/401） | 信号 `tile_selected(tid,name)`/`layer_changed(li)` / `get_tile()`/`get_layer()`/`get_layer_name(li)`/`set_tile()/set_layer()` |
| `EntityPalette.gd` | 左面板·实体 | 6 大类 20 个官方实体模板（NPC/Enemy/Chest/Checkpoint/Portal/Trigger），每个模板带 `{id, kind, name, default_props}` | 信号 `place_requested(id,kind,name,props,pos)` / `find_template(id)` / `all_entity_ids()` |
| `EntityInspector.gd` | 右面板·属性 | 按实体 `kind` 动态生成字段编辑器（共35+字段：HP/ATK/对话/战利品/传送目标等）；支持元数据 JSON 同步 | `bind_entity(node)` / `clear()` / `flush_to_entity()` / 信号 `entity_updated()` |
| `ObjectivesEditor.gd` | 右面板·目标 | 目标配置器：AND/OR/NOT 组合逻辑树 + 10 种目标处理器（REACH_AREA/DEFEAT_ENEMY/PICKUP_ITEM/...） | `load_objectives(arr)` / `export_json()` / 信号 `objectives_updated()` |
| `MapSchemaValidator.gd` | 数据校验 | 程序级 Schema 校验（与 `config/schemas/map.schema.json` 等价）：26 条规则覆盖必选字段、类型、范围、引用一致性 | `validate(map_dict) -> Array[String]`（空数组=通过） |
| `MapSerializer.gd` | 序列化 | `.map.json` 格式统一读写：版本号+8层cells数组+entities数组+objectives数组+meta；自动备份 `*.map.json.bak`；原子写入（先 tmp 后 rename） | `save_to_file(dict, path)` / `load_from_file(path) -> Dictionary` |
| `MapLoader.gd` | 反序列化·实例化 | 游戏端把 .map.json 变成真实场景节点：构建 TileSet + 实例化 8 层 TileMap + 生成 StaticBody2D 统一地板 + 按 kind 实例化 6 类实体 + 注入目标到 LevelFlow | `load_map_to_parent(parent_node, map_dict, inject_objectives=true) -> Dictionary{layers_instantiated,entities_instantiated,cells_count,error_count}` |
| `MapSchemaValidatorHeadless.gd` | 无头验收 | Phase2 自动化验收脚本：`extends SceneTree`，无窗口启动 Godot 直接跑 Validator 校验 3 张官方模板，返回 exit_code 0/1 | （`_init()` 里执行全部逻辑并 quit） |
| `SmokeTestMapIO.gd` | 冒烟测试 | Phase4 Save→Load→Instantiate 链路冒烟：建空地图→存盘→读回→实例化→校验 layers=8 且 errors=0，保证后续编辑器版本不出回归 | （同上，无头模式） |
| `EditorMain.gd` | 编辑器主控制 | 挂 `scenes/editor/EditorMain.tscn`：三栏布局（左面板/中画布/右面板）管理；集成 DrawTools + TilePalette + EntityPalette；Ctrl+S/Ctrl+Z/Y 快捷键；工具栏（新建/保存/模板/测试） | `new_map(cols,rows)` / `load_map(path)` / `save_current()` / `launch_test_current()` |
| `WorkshopMain.gd` | 工坊UI控制 | 挂 `scenes/workshop/WorkshopMain.tscn`：4 大入口（新建/编辑/官方模板/测试用户地图）；扫描 `%APPDATA%/Game1/UserMaps` 目录；自动缺失时重建 3 官方模板 | `user_maps_dir()` / `_workshop_ensure_templates_pub()` / `_on_new_clicked()` / `_on_official_clicked()` |

## 三、迭代演进详解（V0.0 → V0.1 → V0.2）

> 每个迭代按三段式对比：**新增业务特性** → **架构层新增的通用能力** → **架构如何变化/扩展以支持特性**。

---

### 3.1 迭代 V0.1：脚手架 + 配置骨架版

**版本目标**：从零创建 Godot 项目，把后续所有业务依赖的「地基」全部铺好（配置/存档/输入/全局事件/碰撞骨架），无游戏玩法但全部验收用例通过。

#### 3.1.1 V0.1 新增业务特性

| 编号 | 特性 | 验收标准 |
|---|---|---|
| F1.1 | 四层配置可查询 | dotPath 20 项随机查询（跨 L1/L2/L3/L4），0 错误 |
| F1.2 | ProgressFlags 序列化 | 写入 10 个布尔 + 5 个 KV → 序列化 → 反序列化 → 100% 值一致 |
| F1.3 | 3 槽存档读写 | 3 个槽独立写读，时间戳正确，损坏存档返回明确错误 |
| F1.4 | 输入骨架 | 键盘+手柄共 5 核心动作（移动/跳/攻/格/交互）均有控制台日志，无冲突 |
| F1.5 | 碰撞骨架 | V01_InputCollisionTest 场景角色落地不穿墙，平台单向通过 |
| F1.6 | 主菜单占位 | 4 按钮（新游戏/继续/设置/退出），点击有响应 |

#### 3.1.2 V0.1 架构新增的通用能力

| 能力类别 | 新增通用能力 |
|---|---|
| 全局单例机制 | 6 个 Autoload 注册规范，按初始化顺序排序（Config→Flags→Saves→GameEvents→InputBus→LevelFlow） |
| 配置解耦 | Config dotPath 查询 + Layer 枚举 + 用户配置与默认配置自动 Merge |
| 存档分槽 | SaveGameDTO 三层结构 + 校验和，「业务代码只改 body」规范 |
| 输入归一化 | InputBus 中间层，后续加新手柄/改键位无需改业务逻辑 |
| 事件驱动 | GameEvents 总线模式，跨模块零引用通信 |
| 启动流程 | Bootstrap 场景接管初始化顺序，避免 _ready() 竞态 |
| 测试脚手架 | `scenes/test/` 5 个 V01_*Test 场景，按场景编号一键验收 |
| 版本化交付 | `cmd/v0.1/` 目录建立批处理 + 手册规范 |

#### 3.1.3 V0.1 架构如何变化以支持特性

```
V0.0（空目录）
  │
  ▼
① 创建 project.godot 基础配置（渲染器 Forward+、1920×1080、重力 980、物理 60Hz）
  │
  ▼
② 建立 6 层目录骨架（autoload / scenes / scripts / config / resources / cmd）
  │
  ▼
③ 实现 Layer 4（Autoload）：
    ConfigManager → ProgressFlags → SaveSlotManager 串联（Saves 内部序列化 Flags）
    + GameEvents（空信号集，供后续扩展）+ InputBus（基础动作映射）
  │
  ▼
④ 实现 Layer 2（Config）：L1~L4 共 5 JSON 文件，ConfigManager 用 merge 算法实现高层覆盖低层
  │
  ▼
⑤ 实现 Layer 5（Scenes）：
    bootstrap → main_menu → test（5 个测试场景+脚本）
    每个测试场景对应一个 Autoload 能力的验证
  │
  ▼
⑥ 实现 Layer 6（交付）：cmd/v0.1 + doc/V0.1 文档与手册
```

**关键设计决策**：
- 选择「4 层配置」而非单一大 JSON：平衡策划/用户/引擎常量修改权限互不干扰
- 选择「InputBus 中间层」而非直接用 `Input.is_action_pressed()`：手柄适配与冷却防抖集中处理
- 选择「ProgressFlags 双 API」（Set/SetKV）：布尔用于章节进度，KV 用于复杂状态（存路径、槽位、数值），避免 V0.2 存编辑器路径时 API 不兼容

### 3.2 迭代 V0.2：地图编辑器 + 工坊前置版

**版本目标**：对外发布独立地图编辑器（EXE）和游戏内「⚒ 地图工坊」入口，开启社区参与——后续章节地图由编辑器产出而非手搓场景。CharacterBase 与 LevelFlow 前置，为 V0.3 角色战斗/关卡做准备。

#### 3.2.1 V0.2 新增业务特性

| 编号 | 特性 | 验收标准 |
|---|---|---|
| F2.1 | 独立地图编辑器 | 三栏布局，Tile 面板 5 套图集（1~404 共 30 图块）+ 8 图层；Entity 面板 6 大类 20 模板；右面板属性+目标；笔刷/橡皮/矩形填充 |

| F2.3 | `.map.json` 统一格式 | 3 张官方模板 100% 通过 MapSchemaValidator（0 error） |
| F2.4 | JSON Schema 规范 | `config/schemas/map.schema.json` 覆盖 26 条结构规则（必填/类型/范围/引用） |
| F2.5 | 游戏内「工坊」入口 | 主菜单点 ⚒ → 4 按钮（新建/编辑/官方/测试）→ 点击官方模板自动复制到用户目录并打开编辑器 |
| F2.6 | 3 张官方模板 | empty/farm/arena 三张模板均可一键测试：加载→实例化→layers=8→entities 无报错 |

#### 3.2.2 V0.2 架构新增的通用能力

| 能力类别 | 新增通用能力 |
|---|---|
| 地图数据序列化 | `.map.json` 格式标准 + MapSerializer 原子写入（tmp+rename）+ 自动 .bak 备份 |
| 数据合法性校验 | 双轨校验：程序级 MapSchemaValidator（运行时无 JSON Schema 依赖）+ 文件级 map.schema.json |

| 编辑器 UI 模式 | 三栏布局抽象（Palette→Canvas→Inspector），后续加新工具只需加 Palette/Inspector 子类 |
| 无头测试模式 | `extends SceneTree` 脚本模式：无窗口跑 Validator/MapIO 冒烟，供 CI/CD 接入 |
| 全局标志类型扩展 | ProgressFlags 增加 SetKV/GetKV（KeyValue 版），支持存字符串路径/槽位等复杂值 |
| 碰撞解耦 | MapLoader 新增 SOLID_WALL_TILE_IDS 白名单：只有建筑/墙体类加碰撞，地面统一用 StaticBody2D，避免角色卡入地面 |
| 玩家查找三级 fallback | 敌人 AI 三级查找：组("player")→路径("World/Player")→递归 find_child + 冷却，防止场景切换时玩家引用失效 |
| 渲染变换隔离 | `draw_set_transform` 临时变换模式：绘制特效不修改节点自身 transform，避免与物理/朝向状态冲突 |

|---|---|---|
| 选择 `.map.json` 而非 `.tscn` 作为地图格式 | Godot `.tscn` 带内部 ID，版本升级/合并冲突极难处理；JSON 可读可 diff | 社区贡献者可直接 PR 地图文件 |

| ProgressFlags 扩展 SetKV（而非合并成一种 API） | V0.1 只有 Set 布尔版，V0.2 工坊要存 `editor_load_on_start`（字符串路径）和 `slot_index`（int） | 保持向后兼容（旧 `Set/Get` 全部仍有效），新增 KV 专门处理非布尔值 |

## 六、迭代目录规划与 One Track 开发规范

### 6.1 目录规划总原则（One Track 核心 + 6 条不变约束）

> **One Track 模式定义**：全生命周期只用 `main` 一条分支，不建 `feature/*` / `hotfix/*` / `release/*` 多分支；
> 每个迭代结束 → 提交 main → 打 tag `v0.x` → 下一个迭代直接在 main 上继续。
> 通过「**版本化目录永不覆盖 + 提交粒度绑定验收**」保证可追溯、可回退、无合并地狱。

#### One Track 目录 6 条不变约束（迭代之间不会被删除/改名）

| 约束编号 | 规则 | 为什么 |
|---|---|---|
| C1 | **根目录结构固定**：`autoload / scenes / scripts / config / resources / cmd / doc / user_maps` 8 个顶级目录 + `project.godot`，迭代之间**不新增不改名不删除** | 新同学入职 10 秒能找到任何模块；CI/CD 路径写死不用改 |
| C2 | **cmd/ 下按版本建子目录且永不覆盖**：`cmd/v0.1/`、`cmd/v0.2/`、`cmd/v0.3/`… 每个迭代一个子目录，旧目录永远不动 | 任何时候都能跑 cmd/v0.1/03-自动化验收V0.1.cmd 复现 V0.1 的行为，不会被 V0.2 改坏 |
| C3 | **doc/ 下设计文档 + 用户手册双写**：每个迭代同时有 `V0.X_详细设计文档_*.md` 和 `V0.X_用户手册_*.md`，两份并存，永不覆盖旧版 | 设计追溯（设计文档） vs 用户使用（手册）分开；验收时能对比「文档承诺」 vs 「实际交付」 |
| C4 | **scenes/test/ 下按版本号命名验收场景**：`V01_*Test.tscn` / `V02_*Test.tscn` … 旧版验收场景永远保留 | 升级 V0.3 后还能跑 V01_InputCollisionTest.tscn 验证碰撞骨架没被回归破坏 |
| C5 | **config/ 分层目录永不合并**：`L1/L2/L3/L4` 永远分开；`schemas/` 永远独立在 config 下 | 策划改 L2 平衡数值不会误动 L1 常量；校验 Schema 不会和关卡配置混 |
| C6 | **scripts/<模块域>/ 永远按业务域划分子目录，不在 scripts 根堆文件**：V0.1 有 `scripts/config/` `scripts/save/`，V0.2 加 `scripts/editor/`，V0.3 加 `scripts/combat/`、`scripts/characters/`，每个域独立子目录 | 100 个脚本时也不会乱；编辑器模块和战斗模块天然物理隔离 |

### 6.4 One Track 开发规范（落地执行规则）

#### 6.4.1 分支策略（One Track = main 唯一分支）

| 规则 | 具体做法 | 为什么 |
|---|---|---|
| **唯一分支** | 永远只有 `main` 一条分支；不建 `feature/editor` `hotfix/collision` `release/v0.2` | 没有 merge，就没有 merge conflict；小步提交比大分支合并更安全 |
| **打 tag 不建 release 分支** | 每个迭代验收通过 → `git tag v0.2.0` → 继续在 main 开发 V0.3 | 需要回退时 `git reset --hard v0.1.0` 就好，不用维护多分支 |
| **禁止 `git push -f` 公共仓库（若有）** | 本地可以 `reset` 调整，但 push 到远端（如果以后有）必须是 fast-forward | 避免覆盖队友提交 |

#### 6.4.2 提交粒度 = 每个可独立验证的小改动一次提交

> One Track 没有 feature 分支，因此**提交粒度就是回退粒度**。提交太大 = 想回退 1 个小 bug 却要连带回退 10 个功能。

✅ 正确提交（每提交一个单独点，`git log` 一眼看懂）：

#### 6.4.3 版本化目录双写约定（C2 + C3 落地）

每个迭代开始**第一天**必须先做的 3 件事：
```powershell
# 1. 复制 cmd/v0.2 → cmd/v0.3（从一个稳定版本复制起步，不要空写）
Copy-Item -Recurse cmd/v0.2 cmd/v0.3
# 2. 把 v0.3 里的所有 .cmd 文件里的 "V0.2" 替换成 "V0.3"
# 3. 在 doc/ 下新建两个空文档：
#    doc/V0.3_详细设计文档_*.md
#    doc/V0.3_用户手册_*.md
```

> **目的**：迭代第 1 天就把交付物的「坑位」占好，写代码时每做完一个功能立刻把文档和 cmd 脚本填上，不会到验收前夜才补文档。

#### 6.4.4 回退策略（One Track 下必须掌握的 3 种回退）

| 回退场景 | 操作 | 对应目录是否还原 |
|---|---|---|
| **最近 1 次提交写错了，还没 push** | `git reset --soft HEAD^` → 修改 → 重新提交 | ✅ 所有 8 大目录恢复到上一提交状态 |
| **最近 3 次提交改坏了，想整体回到 v0.2.0 tag** | `git reset --hard v0.2.0` | ✅ 整个项目（包括 cmd/v0.1/、doc/V0.1/）**原封不动**回到 tag |
| **只想要回 V0.1 的某个具体文件（比如 ProgressFlags.gd 改坏了，想参考 V0.1 写法）** | `git show v0.1.0:autoload/ProgressFlags.gd > tmp_v01_flags.gd` → 手动对比复制 | ✅ 不会覆盖当前文件，只是导出参考 |

#### 6.4.5 一键保存脚本 `_gitsave.cmd`（One Track 配套工具）

项目根已存在 `_gitsave.cmd`，每次开发中途暂停 / 一个功能做完，直接双击运行：
```batch
@echo off
title GitSave - One Track Auto Commit
git add -A
git --no-pager diff --cached --stat
set MSG=auto-save %date% %time:~0,5%
git commit -m "%MSG%"
git --no-pager log -3 --oneline
```
> 作用：One Track 不要求每次都写规范的 commit message，但要求**每天至少 3 次提交**（早休/午休/下班），用 `_gitsave.cmd` 保证代码不丢，周末想回退到周三下午的某个点永远找得到。
